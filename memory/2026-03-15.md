# 2026-03-15

- Traced the repo's "auto-research" naming ambiguity.
- `llm-backend/autoresearch/` is for downstream tutor prompt optimization using existing `teaching_guidelines`; it does not generate topics/guidelines from books.
- Actual topic extraction and guideline generation pipeline lives in `llm-backend/book_ingestion_v2/`: processing route -> topic extraction orchestrator -> chunk processor -> chapter finalization -> topic sync.
- Current prompt files are already scope-only (`what to teach`, not `how to teach`), so `docs/feature-development/book-ingestion-v2/GUIDELINES_EXTRACTION_AUDIT.md` is partially stale relative to code.
- `autoresearch/book_ingestion_quality/evaluation/` is the experiment/eval harness for the ingestion pipeline itself; it runs the real extraction pipeline or loads existing topics, then scores granularity, coverage depth, and copyright safety.
- Tutor autoresearch's "loop forever" behavior lives in `autoresearch/tutor_teaching_quality/program.md` for an external coding agent; `run_experiment.py` itself is a single-shot runner that reports `pending`/`crash`, not a self-iterating daemon.
- Tutor evaluation entry points differ: the FastAPI evaluation API reads evaluator/simulator model config from DB `llm_config`, while CLI `autoresearch.tutor_teaching_quality.run_experiment` uses env/default `EvalConfig` values unless `--provider` overrides them.
