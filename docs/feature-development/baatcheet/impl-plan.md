# Tech Implementation Plan: Baatcheet — Conversational Teach Me Mode

**Date:** 2026-04-25
**Status:** Draft (revision 2 — incorporates cross-review feedback)
**PRD:** `docs/feature-development/baatcheet/PRD.md` (PR #119)
**Author:** Tech Impl Plan Generator + Manish Jain

---

## 1. Overview

Baatcheet adds a conversational variant of the existing pre-computed Teach Me ("Explain") flow. The student watches a pre-scripted dialogue between the tutor (Mr. Verma) and a peer (Meera), card-by-card, with the same carousel + check-in dispatch already used by Explain. Two new ingestion stages produce the dialogue text (5b) and Baatcheet-specific PixiJS visuals (5c). Stage 10 (audio synthesis) is extended to select voice per `card.speaker`. A new `topic_dialogues` table mirrors the `topic_explanations` storage pattern (single row per guideline; cascade-deleted with the guideline). The frontend gains a new `dialogue_phase` sessionPhase, a Teach-Me sub-chooser, server-side resume-from-last-card (also added to Explain), and a runtime TTS pre-fetch for cards flagged `includes_student_name`.

**Key architectural decisions in this revision:**
1. **Baatcheet is a Teach Me submode**, not a top-level mode. Add `sessions.teach_me_mode = "baatcheet" | "explain"`, keep `sessions.mode = "teach_me"`. Avoids ripple through report card / scorecard / coverage logic that already keys off `mode == "teach_me"`.
2. **Staleness via `source_content_hash`**, not timestamp comparison. `topic_explanations` has only `created_at` (not `updated_at`), and audio/visual stages mutate `cards_json` in place — timestamp comparison would over-trigger after every audio refresh. Hash over semantic fields (lines/content) only.
3. **`_BANNED_PATTERNS` regex enforced in Stage 5b validation**, audio_text_review NOT extended to dialogues by V1. Optional admin button for opt-in dialogue audio review when editor spots trouble.
4. **Extract shared deck behavior from ChatSession.tsx before adding `BaatcheetViewer`** — ChatSession is ~1700 lines and fragile; the refactor pays for itself.
5. **Single `GET /sessions/teach-me-options` aggregator endpoint** for the chooser, instead of inferring availability client-side from multiple existing endpoints.
6. **Welcome card template prepended server-side**, not generated by the LLM — PRD §FR-14 specifies the exact text and LLMs drift even with strict prompts.
7. **Coverage propagation for Baatcheet completion** uses variant A concepts, so report card / scorecard work correctly when student picks Baatcheet.

**Guiding principle:** every new piece of code mirrors an existing one. We are duplicating-and-tweaking, not inventing.

## 2. Architecture Changes

### 2.1 Pipeline DAG (orchestrator)

```
BEFORE                                  AFTER
───────                                 ─────
Layer 1: explanations                   Layer 1: explanations
Layer 2: visuals                        Layer 2: baatcheet_dialogue   ← NEW (depends only on variant A)
Layer 3: check_ins                      Layer 3: baatcheet_visuals    ← NEW (depends on dialogue)
Layer 4: practice_bank                  Layer 4: visuals
Layer 5: audio_review                   Layer 5: check_ins
Layer 6: audio_synthesis                Layer 6: practice_bank
                                        Layer 7: audio_review         (variant A only — unchanged)
                                        Layer 8: audio_synthesis      (extended for dialogues)
```

**Decision: sequential layers, not parallel branch.** PRD §6 frames 5b/5c as a "parallel branch alongside variant A enrichment." In the existing orchestrator, that requires either a per-target lock channel (`ChapterJobService.acquire_lock` is currently keyed `(chapter_id, guideline_id)` and serializes the whole topic) or breaking the single-row contract. Both are scope-creep beyond V1. Sequential adds ~30–60s wall-clock per topic for cold ingestion — acceptable since ingestion is offline, parallelism across topics already exists in `run_chapter_pipeline_all`, and the PRD's <30s success criterion is achievable in V2 by adding a `lock_channel` column. Documented as a follow-up below (§11).

**Decision: 5b/5c slot in BEFORE existing 6a/6b**, not after. Baatcheet generation only depends on variant A explanations (Stage 5), not on visuals or check-ins. Putting 5b/5c before the variant A enrichment lets dialogue work proceed even if visual review fails for variant A. (Variant A enrichment branch may produce its own stale-dialogue signal because variant A's `cards_json` mutates in place, but the content-hash approach in §4.2 makes that signal precise.)

**Decision: audio_text_review stays variant-A-only in V1.** Stage 5b's validators enforce the same `_BANNED_PATTERNS` regex that audio_text_review uses (markdown bold, naked equals, emoji ranges) — the deterministic defects are caught at generation time. Subtle defects that only an LLM reviewer would catch (stage directions, awkward TTS pacing) are not blocked in V1; an opt-in admin button gives editors a safety valve. V2 can flip the default after pilot data shows the actual defect rate.

## 3. Database Changes

### 3.1 New table

| Table | Purpose | Key Columns |
|-------|---------|-------------|
| `topic_dialogues` | Single dialogue per guideline (Baatcheet content) | `id` (UUID PK), `guideline_id` (FK→teaching_guidelines, CASCADE, **unique**), `cards_json` (JSONB), `generator_model` (VARCHAR), `source_variant_key` (VARCHAR, default 'A'), **`source_explanation_id` (VARCHAR, nullable, FK→topic_explanations.id)**, **`source_content_hash` (VARCHAR, nullable)**, `created_at`, `updated_at` |

**Decision: one row per guideline, no `variant_key` in unique constraint.** PRD §5 explicitly excludes multiple dialogue variants per topic. The unique constraint is on `guideline_id` alone — simpler than the explanation pattern. We keep `source_variant_key` as a column (not a unique key) so future expansion to "dialogue derived from variant B" is possible without a schema migration.

**Decision: add `source_content_hash` and `source_explanation_id` (Codex-derived).**
- `source_content_hash`: hash computed at Stage 5b time over the SEMANTIC fields of the source `TopicExplanation.cards_json` (specifically `card_type`, `title`, `lines[].display`, `lines[].audio`, `content`, `audio_text`, optionally `summary_json`). Excludes `audio_url`, `pixi_code`, `visual_explanation` — fields that mutate when only enrichment refreshes. Stale check is `current_hash(variant_a) != stored_hash` instead of timestamp comparison.
- `source_explanation_id`: nullable FK to the `topic_explanations.id` row that produced this dialogue. Pure debuggability — lets editors trace which explanation row generated a given dialogue.

### 3.2 Modified tables

| Table | Change | Details |
|-------|--------|---------|
| `sessions` | **Add column `teach_me_mode VARCHAR DEFAULT 'explain'`** | Distinguishes the two Teach Me submodes within `mode == "teach_me"`. Existing rows backfill to `"explain"`. Practice / clarify_doubts rows are unaffected (column is irrelevant to them). |
| `sessions` | **Rebuild unique index** | Old: `(user_id, guideline_id, mode) WHERE is_paused = TRUE`. New: `(user_id, guideline_id, mode, teach_me_mode) WHERE is_paused = TRUE`. Allows a paused Baatcheet and a paused Explain session for the same topic to coexist (PRD §FR-4). |
| `sessions` | New non-unique lookup index | `(user_id, guideline_id, mode, teach_me_mode, updated_at DESC)` for the resume CTA query (last-touched session per submode). |
| `topic_explanations` | None | Schema unchanged. Stage 10 voice routing uses card-level `speaker` flag in JSONB (no schema change). |
| `chapter_processing_jobs` | None | New `V2JobType` values are stored in the existing `job_type` VARCHAR column. The `idx_chapter_active_topic_job` partial unique index already enforces per-topic serialization. |

### 3.3 Relationships

```
teaching_guidelines ──1:N──► topic_explanations  (existing)
teaching_guidelines ──1:1──► topic_dialogues     (NEW; single row per guideline)
topic_explanations  ──1:0..1──► topic_dialogues  (via source_explanation_id, nullable FK; debug-only)
```

Cascade behavior: deleting a `TeachingGuideline` deletes its `TopicDialogue` (PRD §FR-54). Implemented via `ondelete="CASCADE"` on the FK from `topic_dialogues.guideline_id`. The `source_explanation_id` FK does NOT cascade — it's a debug pointer; if the explanation is regenerated (delete+insert), the FK becomes stale, which is an explicit signal that staleness is also reflected in the content_hash.

### 3.4 Migration plan

Two new helpers in `db.py` following the established pattern.

```python
def _apply_topic_dialogues_table(db_manager):
    """Verify topic_dialogues table exists (created by Base.metadata.create_all)
    and seed any required indexes. Idempotent."""
    inspector = inspect(db_manager.engine)
    if "topic_dialogues" not in inspector.get_table_names():
        logger.warning("topic_dialogues table missing after create_all")
        return
    existing_indexes = {idx["name"] for idx in inspector.get_indexes("topic_dialogues")}
    with db_manager.engine.connect() as conn:
        if "idx_topic_dialogues_guideline" not in existing_indexes:
            conn.execute(text(
                "CREATE UNIQUE INDEX IF NOT EXISTS idx_topic_dialogues_guideline "
                "ON topic_dialogues(guideline_id)"
            ))
        conn.commit()


def _apply_sessions_teach_me_mode_column(db_manager):
    """Add `teach_me_mode` column + rebuild paused-session unique index. Idempotent."""
    inspector = inspect(db_manager.engine)
    if "sessions" not in inspector.get_table_names():
        return
    existing_columns = {c["name"] for c in inspector.get_columns("sessions")}
    with db_manager.engine.connect() as conn:
        if "teach_me_mode" not in existing_columns:
            conn.execute(text(
                "ALTER TABLE sessions ADD COLUMN teach_me_mode VARCHAR DEFAULT 'explain'"
            ))
            # Backfill existing teach_me sessions explicitly
            conn.execute(text(
                "UPDATE sessions SET teach_me_mode = 'explain' "
                "WHERE mode = 'teach_me' AND teach_me_mode IS NULL"
            ))
        # Rebuild paused-session unique index to include teach_me_mode
        conn.execute(text("DROP INDEX IF EXISTS idx_sessions_one_paused_per_user_guideline"))
        conn.execute(text(
            "CREATE UNIQUE INDEX IF NOT EXISTS idx_sessions_one_paused_per_user_guideline "
            "ON sessions(user_id, guideline_id, mode, teach_me_mode) WHERE is_paused = TRUE"
        ))
        conn.execute(text(
            "CREATE INDEX IF NOT EXISTS idx_sessions_user_guideline_teach_mode "
            "ON sessions(user_id, guideline_id, mode, teach_me_mode, updated_at DESC)"
        ))
        conn.commit()
```

Hook them into `migrate()` immediately after `_apply_topic_explanations_table`. Also seed an LLM config row for `baatcheet_dialogue_generator` (provider `claude_code`, model `claude-opus-4-7`) inside `_seed_llm_config()` — same shape as the existing `check_in_enrichment` and `practice_bank_generator` rows. No data backfill is required for `topic_dialogues` (new table). Sessions backfill is the one-line UPDATE above.

class TopicDialogue(Base):
    """Pre-computed Baatcheet dialogue for a teaching guideline.

### 4.2 Repository layer

#### `llm-backend/shared/utils/dialogue_hash.py`

Tiny utility module — content-hash computation, used by both Stage 5b (when storing dialogues) and the pipeline status service (when computing staleness).

```python
import hashlib
import json
from typing import Iterable

# Fields that contribute to the semantic identity of a variant A explanation.
# Excludes audio_url, pixi_code, visual_explanation — these mutate when only
# enrichment refreshes and should NOT cause dialogues to look stale.
_SEMANTIC_LINE_FIELDS = ("display", "audio")
_SEMANTIC_CARD_FIELDS = ("card_type", "title", "content", "audio_text")
_SEMANTIC_SUMMARY_FIELDS = ("key_analogies", "key_examples", "teaching_notes")

def compute_explanation_content_hash(cards_json: list[dict], summary_json: dict | None) -> str:
    """Stable hash of variant A's semantic content. Stage 5b stores this on the
    dialogue row; staleness is `current(variant_A) != stored_hash`."""
    canonical: list[dict] = []
    for card in cards_json or []:
        c = {k: card.get(k) for k in _SEMANTIC_CARD_FIELDS}
        c["lines"] = [
            {k: line.get(k) for k in _SEMANTIC_LINE_FIELDS}
            for line in (card.get("lines") or [])
        ]
        canonical.append(c)
    summary_canonical = (
        {k: (summary_json or {}).get(k) for k in _SEMANTIC_SUMMARY_FIELDS}
        if summary_json else None
    )
    payload = json.dumps(
        {"cards": canonical, "summary": summary_canonical},
        sort_keys=True, separators=(",", ":"),
    )
    return hashlib.sha256(payload.encode("utf-8")).hexdigest()
```

#### `llm-backend/shared/repositories/dialogue_repository.py`

```python
class DialogueRepository:
    """CRUD for topic_dialogues. Mirrors ExplanationRepository.

    Lives in shared/ so both ingestion (book_ingestion_v2) and tutor runtime
    can read it without cross-module imports.
    """

    def __init__(self, db: DBSession):
        self.db = db

    def get_by_guideline_id(self, guideline_id: str) -> Optional[TopicDialogue]:
        return (
            self.db.query(TopicDialogue)
            .filter(TopicDialogue.guideline_id == guideline_id)
            .first()
        )

    def upsert(
        self,
        guideline_id: str,
        cards_json: list[dict],
        generator_model: str,
        source_variant_key: str = "A",
        source_explanation_id: Optional[str] = None,
        source_content_hash: Optional[str] = None,
    ) -> TopicDialogue:
        """Delete-then-insert. Same pattern as ExplanationRepository.upsert."""
        self.db.query(TopicDialogue).filter(
            TopicDialogue.guideline_id == guideline_id
        ).delete()
        d = TopicDialogue(
            id=str(uuid4()),
            guideline_id=guideline_id,
            cards_json=cards_json,
            generator_model=generator_model,
            source_variant_key=source_variant_key,
            source_explanation_id=source_explanation_id,
            source_content_hash=source_content_hash,
        )
        self.db.add(d)
        self.db.commit()
        self.db.refresh(d)
        return d

    def delete_by_guideline_id(self, guideline_id: str) -> int:
        count = (
            self.db.query(TopicDialogue)
            .filter(TopicDialogue.guideline_id == guideline_id)
            .delete()
        )
        self.db.commit()
        return count

    def has_dialogue(self, guideline_id: str) -> bool:
        return (
            self.db.query(TopicDialogue.id)
            .filter(TopicDialogue.guideline_id == guideline_id)
            .first()
        ) is not None

    def is_stale(self, guideline_id: str) -> bool:
        """True iff variant A's current semantic content hash differs from the
        hash stored at dialogue generation time. Used for the admin
        stale-dialogue warning (PRD §FR-46).

        Returns False when no dialogue exists or no variant A exists — those
        states are surfaced separately by the pipeline status service.
        """
        from shared.utils.dialogue_hash import compute_explanation_content_hash
        from shared.models.entities import TopicExplanation

        dialogue = self.get_by_guideline_id(guideline_id)
        if not dialogue or not dialogue.source_content_hash:
            return False
        variant_a = (
            self.db.query(TopicExplanation)
            .filter(
                TopicExplanation.guideline_id == guideline_id,
                TopicExplanation.variant_key == "A",
            )
            .first()
        )
        if not variant_a:
            return False
        current = compute_explanation_content_hash(variant_a.cards_json, variant_a.summary_json)
        return current != dialogue.source_content_hash

    @staticmethod
    def parse_cards(cards_json: list[dict]) -> list[DialogueCard]:
        return [DialogueCard(**c) for c in cards_json]
```

### 4.3 Stage 5b: Dialogue Generation Service

#### `llm-backend/book_ingestion_v2/services/baatcheet_dialogue_generator_service.py`

Mirrors `ExplanationGeneratorService` line-for-line. Key differences:
1. **Welcome card is prepended server-side** after LLM generation, using the literal PRD §FR-14 template — the LLM does not produce card 1.
2. **Validators include `_BANNED_PATTERNS` regex** (markdown bold, naked equals, emoji ranges) on every `lines[i].audio` string — same patterns the existing `audio_text_review_service` uses.
3. **No silent truncation**. If the LLM returns >35 cards (after refine rounds exhausted), the service fails the job. Truncating mid-dialogue produces incoherent decks.
4. **Stores `source_content_hash` and `source_explanation_id`** on upsert.

Key shape:

```python
import re
from shared.utils.dialogue_hash import compute_explanation_content_hash

# Same patterns as audio_text_review_service.py:35-39
_BANNED_AUDIO_PATTERNS = [
    re.compile(r"\*\*"),                                  # markdown bold
    re.compile(r"(?<![a-zA-Z])=(?![a-zA-Z])"),             # naked equals
    re.compile(r"[☀-➿\U0001F300-\U0001FAFF]"),   # emoji
]

WELCOME_CARD_TEMPLATE = (
    "Hi {student_name}! I'm Mr. Verma. Today, Meera is joining us — "
    "she wants to learn about {topic_name} too. Let's start!"
)

class BaatcheetDialogueGeneratorService:
    """Generate the Baatcheet dialogue for a teaching guideline.

    Pipeline: read variant A + guideline → LLM-generate middle dialogue (no welcome)
    → review-refine (N rounds) → prepend literal welcome → validate → store.
    """

    def __init__(self, db: DBSession, llm_service: LLMService):
        self.db = db
        self.llm = llm_service
        self.repo = DialogueRepository(db)
        self.exp_repo = ExplanationRepository(db)
        self.guideline_repo = TeachingGuidelineRepository(db)
        self._generation_schema = LLMService.make_schema_strict(
            DialogueGenerationOutput.model_json_schema()
        )

    def generate_for_guideline(
        self,
        guideline: TeachingGuideline,
        review_rounds: int = 1,
        stage_collector: list | None = None,
        force: bool = False,
    ) -> Optional[TopicDialogue]:
        if not force and self.repo.has_dialogue(guideline.id):
            return None

        variant_a = self.exp_repo.get_variant(guideline.id, "A")
        if not variant_a:
            raise ValueError(f"Variant A not found for {guideline.id}; run stage 5 first")

        guideline_meta = self.guideline_repo._parse_metadata(guideline.metadata_json)
        misconceptions = guideline_meta.common_misconceptions if guideline_meta else []

        # 1. LLM generates the middle (cards 2..N) — no welcome.
        middle_cards = self._generate_middle_cards(guideline, variant_a.cards_json, misconceptions)

        # 2. Review-refine
        for round_num in range(review_rounds):
            middle_cards = self._review_and_refine(middle_cards, guideline, variant_a.cards_json)

        # 3. Prepend the literal welcome template (server-side, not LLM)
        cards = [self._build_welcome_card(guideline)] + middle_cards
        # Re-index cards 1..N
        for i, c in enumerate(cards, start=1):
            c.card_idx = i

        # 4. Validate (raises if invalid — no silent truncation)
        self._validate_cards(cards, raise_on_fail=True)

        # 5. Persist with content hash
        cards_dicts = [c.model_dump() for c in cards]
        for c in cards_dicts:
            c.setdefault("card_id", str(uuid4()))

        content_hash = compute_explanation_content_hash(
            variant_a.cards_json, variant_a.summary_json,
        )
        return self.repo.upsert(
            guideline_id=guideline.id,
            cards_json=cards_dicts,
            generator_model=self.llm.model_id,
            source_variant_key="A",
            source_explanation_id=variant_a.id,
            source_content_hash=content_hash,
        )

    def _build_welcome_card(self, guideline: TeachingGuideline) -> DialogueCardOutput:
        topic_name = guideline.topic_title or guideline.topic
        text = WELCOME_CARD_TEMPLATE  # placeholders resolved at runtime
        return DialogueCardOutput(
            card_idx=1,
            card_type="welcome",
            speaker="tutor",
            speaker_name="Mr. Verma",
            title=None,
            lines=[
                DialogueLineOutput(
                    display=text,
                    audio=text,
                ),
            ],
            includes_student_name=True,  # forces runtime TTS
        )
```

#### Validators — fail-closed

```python
class DialogueValidationError(Exception):
    """Raised when generated cards fail validation. Caller decides whether to
    retry via review-refine or fail the job."""

def _validate_cards(self, cards: list[DialogueCardOutput], *, raise_on_fail: bool) -> bool:
    issues: list[str] = []
    if not (13 <= len(cards) <= 35):
        issues.append(f"card count {len(cards)} outside [13,35]")
    if cards and cards[0].card_type != "welcome":
        issues.append("first card must be welcome")
    if cards and cards[-1].card_type != "summary":
        issues.append("last card must be summary")

    last_check_in = -10
    for i, c in enumerate(cards):
        if c.card_type == "check_in":
            if i - last_check_in < 4:
                issues.append(f"check-ins too close: idx {last_check_in}, {i}")
            last_check_in = i
        if c.card_type in ("tutor_turn", "peer_turn") and not c.lines:
            issues.append(f"card {c.card_idx}: {c.card_type} has no lines")
        if c.card_type == "visual" and not (c.visual_intent or "").strip():
            issues.append(f"card {c.card_idx}: visual missing visual_intent")

        # _BANNED_PATTERNS: no markdown / naked equals / emoji in audio strings
        for li, line in enumerate(c.lines):
            for pat in _BANNED_AUDIO_PATTERNS:
                if pat.search(line.audio):
                    issues.append(
                        f"card {c.card_idx} line {li}: banned pattern in audio "
                        f"({pat.pattern})"
                    )

        # includes_student_name → text MUST contain {student_name}; reverse also checked
        text_blob = " ".join(line.audio for line in c.lines)
        has_placeholder = "{student_name}" in text_blob
        if c.includes_student_name and not has_placeholder:
            issues.append(f"card {c.card_idx}: flagged includes_student_name but no placeholder")
        if has_placeholder and not c.includes_student_name:
            issues.append(f"card {c.card_idx}: contains {{student_name}} but flag not set")

    if issues and raise_on_fail:
        raise DialogueValidationError("; ".join(issues[:8]))
    return not issues
```

The service's review-refine loop catches `DialogueValidationError`, feeds the issues list into the next refine prompt as repair instructions, and retries. After exhausting `review_rounds`, the final validation pass runs with `raise_on_fail=True`, which propagates to `_run_baatcheet_dialogue_generation` and fails the job.

#### `llm-backend/book_ingestion_v2/services/baatcheet_visual_enrichment_service.py`

```python
class BaatcheetVisualEnrichmentService:
    """Stage 5c: fill visual_explanation slots in topic_dialogues using PixiCodeGenerator."""

    def __init__(self, db: DBSession, llm_service: LLMService):
        self.db = db
        self.repo = DialogueRepository(db)
        self.pixi_gen = PixiCodeGenerator(llm_service)

    async def enrich_guideline(
        self, guideline_id: str, force: bool = False,
    ) -> Optional[TopicDialogue]:
        dialogue = self.repo.get_by_guideline_id(guideline_id)
        if not dialogue:
            return None

        cards = list(dialogue.cards_json)  # mutable copy
        modified = False

        for card in cards:
            if card.get("card_type") != "visual":
                continue
            if card.get("visual_explanation") and not force:
                continue
            intent = card.get("visual_intent") or card.get("title") or ""
            if not intent.strip():
                continue
            pixi_code = await self.pixi_gen.generate(intent, output_type="image")
            if not pixi_code:
                continue
            card["visual_explanation"] = {
                "output_type": "static_visual",
                "title": card.get("title"),
                "visual_summary": intent,
                "pixi_code": pixi_code,
            }
            modified = True

        if modified:
            return self.repo.upsert(
                guideline_id=guideline_id,
                cards_json=cards,
                generator_model=dialogue.generator_model,
                source_variant_key=dialogue.source_variant_key,
                source_explanation_id=dialogue.source_explanation_id,
                source_content_hash=dialogue.source_content_hash,
            )
        return dialogue
```

**Decision: no separate review-refine pass for visuals.** `AnimationEnrichmentService` has visual code review for variant A because variant A visuals are core to the explanation. Baatcheet visuals are accents (PRD §15 punts illustrated avatars to V2). Match the simpler check-in-style "generate once, regenerate if needed" pattern. If quality is insufficient post-launch, copy review-refine rounds from `AnimationEnrichmentService`.

```python
def generate_for_topic_dialogue(
    self, dialogue: TopicDialogue, *, dry_run: bool = False,
) -> Optional[list[dict]]:
    """Generate audio for a TopicDialogue record.

    - Skips lines on cards with includes_student_name=True (frontend handles runtime TTS).
    - Routes voice via card.speaker.
    - S3 keys: audio/{guideline_id}/dialogue/{card_id}/{line_idx}.mp3
              audio/{guideline_id}/dialogue/{card_id}/check_in/{field}.mp3
    """
    cards = dialogue.cards_json
    if not cards:
        return None

    guideline_id = dialogue.guideline_id
    for card in cards:
        if card.get("includes_student_name"):
            continue  # Runtime TTS — never pre-render.

        speaker = card.get("speaker")
        lang_code, voice_name = _voice_for_speaker(speaker, self.language)
        voice = texttospeech.VoiceSelectionParams(language_code=lang_code, name=voice_name)
        card_id = card.get("card_id")
        if not card_id:
            logger.warning(f"Dialogue card at idx {card.get('card_idx')} missing card_id; skipping audio")
            continue

        # Explanation-shaped lines (works for tutor_turn, peer_turn, welcome, summary, visual)
        for line_idx, line in enumerate(card.get("lines") or []):
            if line.get("audio_url"):
                continue
            text = (line.get("audio") or "").strip()
            if not text or "{student_name}" in text:
                continue
            s3_key = f"audio/{guideline_id}/dialogue/{card_id}/{line_idx}.mp3"
            try:
                line["audio_url"] = self._synth_and_upload(text, s3_key, voice=voice)
            except Exception as e:
                logger.error(f"Dialogue TTS failed for {guideline_id}/{card_id}/line{line_idx}: {e}")

        # Check-in fields use tutor voice (PRD §FR-27: no Meera reactions in V1)
        check_in = card.get("check_in")
        if check_in and card.get("card_type") == "check_in":
            tutor_voice = texttospeech.VoiceSelectionParams(
                language_code=TUTOR_VOICE[0], name=TUTOR_VOICE[1],
            )
            for text_field, key_suffix, url_field in _check_in_fields_for(check_in):
                if check_in.get(url_field):
                    continue
                text = (check_in.get(text_field) or "").strip()
                if not text:
                    continue
                s3_key = f"audio/{guideline_id}/dialogue/{card_id}/check_in/{key_suffix}.mp3"
                try:
                    check_in[url_field] = self._synth_and_upload(text, s3_key, voice=tutor_voice)
                except Exception as e:
                    logger.error(f"Dialogue check-in TTS failed for {guideline_id}/{card_id}/{key_suffix}: {e}")

    return cards
```

**Decision: do NOT change variant A's S3 key convention.** Existing keys (`audio/{guideline_id}/{variant_key}/...`) keep working. Dialogue uses a parallel namespace `audio/{guideline_id}/dialogue/{card_id}/...`.

**Decision: `card_id` is mandatory for dialogue cards.** Stage 5b assigns one per card during `upsert`. Dialogue cards use `card_id` everywhere because the dialogue may be regenerated; positional keys like `card_idx` would race during regeneration.

#### Coverage / report card propagation

New `_finalize_baatcheet_session(session, user_id)` in `SessionService`:

```python
def _finalize_baatcheet_session(self, session: SessionState, user_id: Optional[str]):
    """Mark Baatcheet session complete + propagate coverage to report card.

    Coverage uses VARIANT A concepts as the canonical source — Baatcheet teaches
    the same material as Explain, so completion should contribute identical
    coverage regardless of which submode the student picked."""
    if not session.dialogue_phase:
        return
    session.dialogue_phase.completed = True
    session.dialogue_phase.active = False
    session.is_paused = False  # explicit completion clears pause

    # Pull covered concepts from variant A summary (same source the existing
    # _finalize_teach_me_session uses for Explain mode).
    variant_a = ExplanationRepository(self.db).get_variant(
        session.dialogue_phase.guideline_id, "A",
    )
    if variant_a and variant_a.summary_json:
        covered = variant_a.summary_json.get("card_titles") or []
        session.covered_concepts = covered
    else:
        # Fallback: dialogue card titles excluding welcome/check_in/summary
        dialogue = DialogueRepository(self.db).get_by_guideline_id(
            session.dialogue_phase.guideline_id,
        )
        if dialogue:
            session.covered_concepts = [
                c.get("title") for c in dialogue.cards_json
                if c.get("card_type") not in ("welcome", "check_in", "summary")
                and c.get("title")
            ]
```

Without this, Baatcheet sessions complete silently and contribute zero coverage to scorecard. Real bug.

### 5.4 Personalized audio with concurrency cap

#### `llm-frontend/src/hooks/usePersonalizedAudio.ts`

```tsx
const MAX_CONCURRENT_TTS = 4;

async function withConcurrencyLimit<T>(
  tasks: Array<() => Promise<T>>,
  limit: number,
): Promise<T[]> {
  const results: T[] = [];
  const executing: Promise<void>[] = [];
  for (const task of tasks) {
    const p = task().then((r) => { results.push(r); });
    executing.push(p);
    if (executing.length >= limit) {
      await Promise.race(executing);
      executing.splice(executing.findIndex((x) => x === p), 1);
    }
  }
  await Promise.all(executing);
  return results;
}

export function usePersonalizedAudio(
  cards: DialogueCard[] | null,
  personalization: Personalization,
  language: string,
) {
  useEffect(() => {
    if (!cards) return;
    const personalized = cards.filter(c => c.includes_student_name);
    if (personalized.length === 0) return;

    const tasks = personalized.flatMap((card) =>
      card.lines.map((line, lineIdx) => async () => {
        const text = materializeText(line.audio, personalization);
        if (!text.trim()) return;
        try {
          const blob = await synthesizeSpeech(text, language, {
            voiceRole: card.speaker || 'tutor',
          });
          attachClientAudioBlob(`personalized:${card.card_id}:${lineIdx}`, blob);
        } catch (e) {
          console.warn('Runtime TTS failed for card', card.card_id, e);
        }
      }),
    );

    withConcurrencyLimit(tasks, MAX_CONCURRENT_TTS);
  }, [cards, personalization.student_name, personalization.topic_name, language]);
}

function materializeText(text: string, p: Personalization): string {
  const name = p.student_name?.trim() || p.fallback_student_name || 'friend';
  return text
    .replaceAll('{student_name}', name)
    .replaceAll('{topic_name}', p.topic_name);
}
```

`attachClientAudioBlob` is a small extension to `audioController.ts` that stores a `Blob` under a synthetic key and surfaces a `blob:` URL the playback path can resolve. Falls back to plain text-only display per PRD §12.

## 6. LLM Integration

### 6.1 New agent: Baatcheet dialogue generator (Stage 5b)

- **Provider:** `claude_code` (subprocess wrapper, mirrors `check_in_enrichment` and `practice_bank_generator`).
- **Model:** `claude-opus-4-7` initially. Configurable via `llm_configs`.
- **Reasoning effort:** `high` (matches Stage 5).
- **Structured output:** `DialogueGenerationOutput` Pydantic schema, validated post-call by the strict validator.
- **Prompt strategy:** split system prompt (static rules + JSON schema, loaded via `--append-system-prompt-file`) and dynamic user prompt (topic + variant A cards + misconceptions). Same split optimization as Stage 5.
- **Review-refine:** 1 round at `balanced`, 2 rounds at `thorough` (default 0 at `fast`). The validator's issue list is fed into the refine prompt as repair instructions.

### 6.2 Reused: PixiJS code generator (Stage 5c)

Reuses `tutor/services/pixi_code_generator.py` unchanged. New prompt template `prompts/baatcheet_visual_intent.txt` is just an input format — `PixiCodeGenerator.generate(prompt, output_type)` takes a plain string. The Baatcheet enrichment service constructs that string from `card.title + " — " + card.visual_intent`.

### 6.3 Cost and latency considerations

- **Stage 5b token usage estimate:** ~3000 input tokens (variant A cards + guideline + misconceptions + system prompt) → ~6000 output tokens (24–29 LLM-generated cards × ~200 tokens). ~30s per round at Opus high effort. At 2 rounds: ~60s wall-clock per topic.
- **Stage 5c token usage:** ~5 visual cards per dialogue × ~500 input + ~2000 output = small. ~30s total per topic.
- **Stage 10 audio (extension):** ~25 lines per dialogue × Google TTS Chirp 3 HD ≈ ~10s per topic. Cost: ~$0.06 per topic. Skips the 3–5 personalized cards (runtime TTS at session start instead).
- **Caching:** structured prompt-cache via Claude Code adapter is automatic for the static system prompt. The dynamic prompt won't cache cross-topic but will cache within review-refine rounds (same topic, same variant A).

## 10. Deployment Considerations

### 10.1 Migration

- `topic_dialogues` table created via `Base.metadata.create_all()`.
- `sessions.teach_me_mode` added via idempotent `ALTER TABLE`.
- Backfill: `UPDATE sessions SET teach_me_mode='explain' WHERE mode='teach_me' AND teach_me_mode IS NULL` (one-shot inside `_apply_sessions_teach_me_mode_column`).
- Paused-session unique index rebuild: idempotent DROP + CREATE.

### 10.2 Order of operations (deploy)

1. **Deploy backend first**: new table, new column, new endpoints, audio service extended. Existing variant A path is fully backwards-compatible (`teach_me_mode` defaults to `"explain"` when omitted).
2. **Deploy frontend** with new UI. Until backend is live, the `teach_me_mode: "baatcheet"` request would 404. Frontend should defensively check `availability` via `getTeachMeOptions` before showing the Baatcheet card.
3. **Run ingestion** (admin manual) for at least one topic to seed dialogue data.
4. **Soft launch** with feature flag `enable_baatcheet_mode = false`. Sub-chooser defaults to invisible (mode-select goes straight to Explain), preserving pre-Baatcheet UX exactly.
5. **Pilot**: flip flag for internal users. Validate completion paths, coverage propagation, audio quality. Monitor TTS API errors.
6. **GA**: flip flag for all users once Baatcheet availability covers sufficient topics.

### 10.3 Rollback plan

- Feature flag `enable_baatcheet_mode = false` makes the sub-chooser invisible. Existing topics still work via Explain path (default `teach_me_mode = "explain"`).
- Backend endpoints are additive. Rolling back just the frontend flag is sufficient for student-facing rollback.
- Data: `topic_dialogues` rows can be left in place (orphan but harmless). Drop the table if doing a full revert.
- Audio: dialogue MP3s under `audio/{guideline_id}/dialogue/...`; cleanup is `aws s3 rm` on that prefix only.
- `sessions.teach_me_mode` column can be left in place — existing sessions ignore it.

### 10.4 Infrastructure

- No new AWS resources. Existing S3, RDS, Cognito reused.
- Google Cloud TTS quota: ~25 lines × N topics × 1.5x retries on cold ingestion. Far below daily quota; monitor existing CloudWatch metrics.

## Appendix A: Decision log

| Decision | Why |
|---|---|
| **`teach_me_mode` submode, not top-level `mode`** | Preserves existing report card / scorecard / coverage logic that keys off `mode == "teach_me"`. Top-level mode change would ripple through analytics, mode routes, paused-session uniqueness. PRD §3 places Baatcheet *under* Teach Me. |
| **`source_content_hash` for staleness, not timestamp comparison** | `topic_explanations` has only `created_at`, no `updated_at`. Even adding `updated_at` would over-trigger because audio/visual stages mutate `cards_json` in place. Hash over semantic fields only captures the actual product intent. |
| **Welcome card prepended server-side, not LLM-generated** | PRD §FR-14 specifies the literal template. LLMs drift even with strict prompts. Service-side prepend is bulletproof. |
| **No silent truncation in Stage 5b** | `ExplanationGeneratorService` truncates at 15; that pattern was inherited unintentionally. For 25–30 card dialogues, mid-deck truncation produces incoherent output. Fail closed; review-refine fixes overproduction. |
| **`_BANNED_PATTERNS` regex in Stage 5b validators (skip default audio_text_review for dialogues)** | Catches deterministic defects (markdown bold, naked equals, emoji) at generation time. Avoids ~+150s ingestion latency that always-on dialogue review would add. Editor opt-in safety valve covers the residual risk. |
| **One row per guideline in `topic_dialogues` (no `variant_key` in unique constraint)** | PRD §5 excludes multiple variants. Single FK index, no composite uniqueness. |
| **Sequential pipeline layers (5b/5c BEFORE variant A enrichment)** | Existing per-topic lock would force serialization anyway. Parallel branch needs `lock_channel` schema change beyond V1. Placing 5b/5c immediately after Stage 5 lets dialogue work proceed even if variant A enrichment fails — clean separation. |
| **Reuse `ExplanationLine` and `CheckInActivity` Pydantic models** | They already encode display+audio split and 11-type check-ins. Inventing parallel models guarantees future drift. |
| **`DialoguePhaseState` separate from `CardPhaseState`** | They share 3 fields but `CardPhaseState` is variant-aware. One struct creates nullable-fields-easy-to-misuse. |
| **Server-side `last_card_idx` via `state.dialogue_phase` + new `/card-progress` endpoint** | `state_json` already mode-flexible. Single richer endpoint (with `mark_complete` and `check_in_events`) reduces state-machine complexity vs the old combination of websocket + `/card-action`. Existing websocket `card_navigate` deprecated. |
| **Voice routing as a per-call parameter (fallback to instance default)** | Variant A cards lack `speaker`; falling through to `self.voice` keeps existing behavior identical. |
| **Dialogue audio S3 keys in a separate prefix `dialogue/`** | Variant A keys unchanged. Backwards compat is mechanical, not contractual. |
| **`card_id` mandatory for dialogue cards** | Dialogue regen rotates content; positional keys race. UUID per card decouples audio identity from order. |
| **Sub-chooser as separate page with `getTeachMeOptions` aggregator** | Server holds full truth. Frontend can't miscompute disabled-state or resume CTAs. |
| **Stage 5b uses Claude Code provider** | Matches existing high-pedagogical-quality stages. Configurable via `llm_configs`. |
| **No review-refine in Stage 5c (visual enrichment)** | Matches simpler check-in pattern. PRD doesn't require it; V2 can add. |
| **TTS endpoint accepts only `voice_role`, not raw voice name** | Security boundary — frontend can't pass arbitrary Google voice IDs. |
| **TTS concurrency cap = 4 in `usePersonalizedAudio`** | Avoids bursting Google TTS rate limits when topic has many personalized cards. |
| **Refresher topics force `teach_me_mode = "explain"` in V1** | PRD doesn't address; safest default. Revisit with product. |
| **Coverage propagation: Baatcheet finalizer uses variant A concepts** | Without this, Baatcheet completion contributes zero coverage. Variant A is canonical source regardless of which submode the student picked. |
| **Refactor ChatSession.tsx before adding `dialogue_phase`** | Adding a third sessionPhase value inline compounds existing tech debt. Extraction into `DeckCarousel`/`ExplanationViewer`/`BaatcheetViewer` benefits both modes. Non-negotiable. |
| **Fix Explain replay off-by-one bug as part of resume work** | PRD §FR-33 scope-expansion implicitly requires it; current code uses `current_card_idx + 1` which double-counts the welcome card. |
