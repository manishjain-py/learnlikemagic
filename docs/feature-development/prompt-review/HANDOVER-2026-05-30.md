## 3. The design (locked decisions — the "why")

Settled through an interview with Manish. Don't relitigate these without him:

1. **Local interactive Claude Code skill** (a command), not cloud/autonomous. Interview is first-class.
2. **Stored KB**, built once, refresh on demand. Weighted to Claude/Anthropic since these prompts run through Claude Code.
3. **Empirical before/after eval is in v1** (not just static reasoning). This is the core safety mechanism — Manish has been burned by losing load-bearing examples when "simplifying" (see memory `feedback_prompt_rule_consolidation`).
4. **Eval is per-prompt, not universal.** Reuse whatever already runs that prompt; generic blind LLM-judge as fallback. Baatcheet is the proving ground (it has runners + datasets + validators).
5. **Discover-fresh each run** — no stored recipe/registry, no per-prompt adapter code. The skill greps for how to run a prompt at runtime. **Zero product code added to `llm-backend/`**; any glue lives in throwaway scratch and is deleted. (This was Manish's explicit concern: don't accumulate code.)
6. **Eval once, at the end**, on the agreed rewrite. Every change is tagged **cosmetic** (can't change output) or **substantive** (could). If the rewrite regresses, bisect by tag: cosmetic-only (no-op floor) vs full → pins damage to the substantive few.
7. **Review the system+user pair together** (cross-file redundancy + assumptions).
8. **Input = file path.** Interview goes **section by section, one cluster per turn, recommendation each time** — not a giant dump, not word-by-word.
9. **Lane:** judges *craft* (clarity/concision/plain language), not pedagogy. Anything encoding a `docs/principles/` rule is protected; principle *contradictions* are flagged to Manish, never silently fixed. Never edits `docs/principles/`. Never auto-commits.

## 5. The eval mechanism (for Phase 3 — fully local, no DB, no file mutation)

Each saved topic dir keeps its **filled** dialogue user-prompt, so we re-generate by feeding that exact input + the system prompt (old vs new). **Two usable inputs:**
- `llm-backend/scripts/baatcheet_v2_outputs/water-cycle-class4/run01-high/02_dialogue_user_prompt.txt` (Science G4, has spine)
- `llm-backend/scripts/baatcheet_v2_outputs/5-6-digit-indian-numbers/run01-high/02_dialogue_user_prompt.txt` (Math, procedural, no spine)
- (`prod-23632b15/` has no saved dialogue user-prompt — would need reconstructing for a 3rd input.)

**How to run a cell** (pattern from `llm-backend/scripts/baatcheet_v2_experiment.py:88-143`):
```
env -u ANTHROPIC_API_KEY claude -p --output-format json \
  --dangerously-skip-permissions --no-session-persistence --max-turns 1 \
  --model claude-opus-4-7 --effort high \
  --append-system-prompt-file <SYSTEM_PROMPT_PATH>   < <02_dialogue_user_prompt.txt>
```
- Write the **candidate** system prompt to `.claude/prompt-review/.scratch/baatcheet-dialogue/candidate_system.txt` and point `--append-system-prompt-file` at it — **never mutate the real prompt file.** ("old" = current real file text copied to scratch too.)
- Parse the JSON envelope's `result`, extract the dialogue JSON (see `extract_json` in the harness).
- **Strip `ANTHROPIC_API_KEY`** from env (critical — see CLAUDE.md / claude_code_adapter).
- Model note: harness uses `opus-4-7`; current session is `opus-4-8`. Pick one, be consistent across old & new.

**Scoring:**
- Mechanical: `python scripts/baatcheet_v2_eval.py <run-dir>` (interjection/sound/spine-callback counts, card-type mix). Reference `eval_scores.json` exists in each saved dir.
- Blind LLM-judge: show the two dialogues unlabeled, randomized order, ask which better serves the prompt goal + `baatcheet-dialogue-craft` + `easy-english` + house-style. 2 runs/cell to cut noise.

**Plan:** inputs = water-cycle + numbers; defer ALL generation until changes agreed (spend nothing if all-cosmetic). Cost ≈ 4–8 Opus generations (2 inputs × old/new × ~2 runs).
