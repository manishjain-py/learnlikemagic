# PRD: Check-In Card Review-Refine

## Problem

Check-in cards are short interactive activities (sort_buckets, tap_to_eliminate, match_pairs, sequence, etc.) inserted between explanation cards. They are generated by a single LLM pass in `CheckInEnrichmentService`, then validated structurally (item counts, index bounds, bucket distribution, no duplicates).

The current validator **never inspects content semantically**. So a card that is structurally perfect but factually wrong ships to students as-is. Observed example:

> **"Put each number in the right group."**
> 3-DIGIT NUMBER: 352, **87**, 999
> 4-DIGIT NUMBER: 7,089, 1,000, 3,527

"87" has two digits, not three. The validator accepted it because the bucket distribution rule was satisfied.

Similar accuracy bugs can slip through across every card type:

| Card type | Accuracy failure |
|-----------|------------------|
| sort_buckets | Item placed under wrong label (the "87" case) |
| sequence | Items marked as "correct order" are actually out of order |
| spot_the_error | The flagged step is actually correct, or a different step is the real error |
| tap_to_eliminate | Two of the N options are actually correct, or the marked correct answer is wrong |
| pick_one / fill_blank | `correct_index` points to a wrong option |
| match_pairs | Pairs are actually mismatched or ambiguous |
| true_false | `correct_answer` is the wrong polarity |
| predict_then_reveal | Reveal text contradicts the marked correct option |
| odd_one_out | Marked odd item is not actually the odd one |

These are trust-breaking for students and tutors: the platform is explicitly telling the student that a factually wrong thing is right.

## Solution

Add a **review-refine stage** to check-in generation, mirroring the one we already run on explanation cards (`ExplanationGeneratorService._review_and_refine`). A second LLM pass sees the generated check-ins alongside the teaching guideline and preceding explanation cards, verifies every factual claim and every marked correct answer, and rewrites cards in place when it finds accuracy problems.

**Scope is deliberately narrow: accuracy only.** The reviewer does not rewrite for tone, distractor plausibility, cognitive level, instruction clarity, or conceptual alignment. It leaves those alone so that good generation output is preserved and the reviewer's job is unambiguous.

Configurable `review_rounds` (0–5, default 1), plumbed through the admin API and exposed as a dropdown on the existing chapter-level admin page, mirroring the existing `ExplanationAdmin` control.

## Goals

- Catch accuracy bugs like the "87 in 3-DIGIT" example before students see them.
- Reuse the review-refine pattern already proven for explanation cards — same prompt structure, same retry loop, same stage-snapshot capture.
- Zero schema changes. Same `CheckInGenerationOutput` returned; structural validator continues to run after refinement.
- Zero regression risk at `review_rounds=0` — the loop is bypassed entirely.

## Non-Goals

- **No clarity/tone/distractor-quality rewrites.** Reviewer is instructed to leave those fields alone. Scope creep on the prompt will make the reviewer unpredictable and undo good generation output.
- **No deterministic rule-based validator.** We chose LLM-only per product direction.
- **No new admin page.** The `review_rounds` selector is added next to the existing "Check-ins" button on `BookV2Detail.tsx`, matching where the action lives today.
- **No student-facing UI changes.** The card shape is unchanged.
- **No API contract changes beyond adding `review_rounds`.** No new endpoints.

## User Experience (Admin)

### Before

On the book detail page, the admin expands a chapter and clicks a single "Check-ins" button. The job runs with no configurable quality knobs.

### After

Next to the "Check-ins" button, a "Review rounds" dropdown appears (0–5, default 1). The selected value is passed to the enrichment job, which runs N review-refine rounds after the initial generation pass.

```
[Explanations] [Refresher] [Visuals] [Check-ins]  Review rounds: [1 ▾]
```

The dropdown mirrors the control already present on the `ExplanationAdmin` page (`ExplanationAdmin.tsx:367–373`), so the admin sees a consistent pattern for both pipelines.

### Observability

`review_rounds=0` is explicitly supported and disables the reviewer entirely — useful as an escape hatch if the reviewer starts making cards worse, and as a baseline for before/after comparisons.

## Success Criteria

1. Running check-in enrichment with `review_rounds=1` on a chapter that previously produced the "87 in 3-DIGIT" bug regenerates cards without that class of accuracy error.
2. The existing structural validator continues to reject any refinement output that breaks structural invariants (item counts, index bounds, bucket distribution, duplicates). Review cannot smuggle structurally broken cards past validation.
3. `review_rounds=0` produces output bit-identical to today's pipeline.
4. No regression in job completion time at `review_rounds=1` for a typical chapter (expect ~1.5–2× LLM cost per variant, acceptable for offline enrichment).

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Reviewer rewrites things it shouldn't (tone, distractors) | Prompt is explicit: accuracy-only scope, leave other fields unchanged. |
| Reviewer introduces structural breakage (wrong item counts, index out of range) | Existing `_validate_check_ins` runs after every refine round; bad outputs get dropped like before. |
| Latency/cost blow-up at high `review_rounds` | Param capped at 5 server-side; default 1; reviewer runs offline in enrichment pipeline, not sync path. |
| Reviewer loop never converges (oscillates between revisions) | Fixed round count (not convergence-based) — always terminates. |
| LLM output shape drift | Same `CheckInGenerationOutput` Pydantic schema; any drift fails validation and the original output is kept. |

## Rollout

- Ship behind the `review_rounds` param (default 1) — opt-in per admin invocation.
- Admin can manually re-run check-in enrichment on a chapter after this lands to get reviewed check-ins.
- No backfill across all books in this PR. Admin can batch-reprocess at their own pace.
