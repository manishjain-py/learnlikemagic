# Issue Reporting System — Implementation Plan

## Overview
Build an issue reporting system that lets testers report bugs while using the app, with LLM-powered issue interpretation, and an admin page for developers to manage issues.

---

## Architecture

### Two New Admin Pages
1. **Report an Issue** (`/admin/report-issue`) — Form with text, voice, screenshots + LLM interpretation loop
2. **Issues List** (`/admin/issues`) — View all issues, filter by status, change status

### Backend Module
New `issues/` module following existing patterns (API → Service → Repository).

---

## Step-by-Step Plan

### Step 1: Database Model + Migration
**File:** `llm-backend/shared/models/entities.py`
- Add `Issue` model:
  - `id` (String, PK, uuid)
  - `user_id` (String, FK → users.id, nullable)
  - `reporter_name` (String, nullable) — denormalized for display
  - `title` (String) — short summary generated by LLM
  - `description` (Text) — LLM-refined issue description
  - `raw_input` (Text) — original user input (text + transcribed voice)
  - `screenshot_urls` (JSONB) — list of S3 URLs
  - `status` (String, default='open') — open/in_progress/closed
  - `created_at`, `updated_at` (DateTime)

**File:** `llm-backend/db.py`
- Add `_apply_issues_table()` migration function
- Seed `issue_interpreter` LLM config entry

### Step 2: Backend — Repository
**File:** `llm-backend/issues/repositories/issue_repository.py`
- `create(issue)` — insert new issue
- `get_by_id(issue_id)` — get single issue
- `list_all(status_filter, offset, limit)` — list with optional status filter
- `update_status(issue_id, status)` — change status
- `count(status_filter)` — count for pagination

### Step 3: Backend — Service + LLM Interpretation
**File:** `llm-backend/issues/services/issue_service.py`
- `interpret_issue(raw_input, screenshots_info, conversation_history)` → returns LLM-interpreted issue text
  - Uses LLMService with component_key `issue_interpreter`
  - Prompt includes app context (what LearnLikeMagic is, common features) so the LLM can understand issues in context
  - Returns a clear, structured interpretation
- `refine_issue(previous_interpretation, user_feedback, conversation_history)` → refined interpretation
  - Iterative refinement based on user feedback
- `create_issue(user_id, reporter_name, title, description, raw_input, screenshot_urls)` → created issue
- `list_issues(status, offset, limit)` → paginated list
- `update_status(issue_id, status)` → updated issue

### Step 4: Backend — API Routes
**File:** `llm-backend/issues/api/issue_routes.py`
- `POST /api/issues/interpret` — Send raw text + screenshot filenames, get LLM interpretation
  - Body: `{ raw_input, screenshot_names?, conversation_history? }`
  - Response: `{ interpreted_title, interpreted_description }`
- `POST /api/issues/refine` — Send feedback on interpretation
  - Body: `{ previous_interpretation, user_feedback, conversation_history }`
  - Response: `{ interpreted_title, interpreted_description }`
- `POST /api/issues/upload-screenshot` — Upload a screenshot file to S3
  - Accepts multipart file upload
  - Returns: `{ url, s3_key }`
- `POST /api/issues` — Create final issue
  - Body: `{ title, description, raw_input, screenshot_urls }`
- `GET /api/issues` — List issues with optional `?status=open` filter
- `PATCH /api/issues/{id}/status` — Update issue status
  - Body: `{ status: "open" | "in_progress" | "closed" }`

**File:** `llm-backend/main.py`
- Register the issues router

### Step 5: Frontend — Report Issue Page
**File:** `llm-frontend/src/features/admin/pages/ReportIssuePage.tsx`

**UI Flow:**
1. **Input Phase:**
   - Large text area for typing the issue
   - Mic button (records audio → sends to existing `/transcribe` endpoint → appends text)
   - Upload button for screenshots (multiple, shows thumbnails)
   - "Interpret" button to submit

2. **Review Phase:**
   - Shows LLM's interpretation (title + description) in a card
   - Two buttons: "Yes, Submit" and "No, let me clarify"
   - If "clarify": shows a feedback text box (+ mic) → re-interprets → shows again
   - This loops until user accepts

3. **Submitted Phase:**
   - Success message with issue ID
   - Button to report another issue or go back

**Voice Recording:**
- Use `MediaRecorder` API in browser to capture audio
- Send blob to existing `POST /transcribe` endpoint
- Append transcribed text to the text area

**Screenshot Upload:**
- File input (accept="image/*", multiple)
- Upload each to `POST /api/issues/upload-screenshot`
- Show thumbnails with remove button

### Step 6: Frontend — Issues List Page

## LLM Prompt Strategy (Issue Interpreter)

The prompt for issue interpretation will:
1. Describe what LearnLikeMagic is (adaptive tutoring platform for students)
2. List the key features (learning sessions, exams, report cards, book ingestion, etc.)
3. Ask the LLM to interpret the user's raw input as a bug report / feature request
4. Output a concise title (1 line) and structured description (what happened, expected behavior, where in the app)

For refinement, include the previous interpretation + user feedback in the conversation.
